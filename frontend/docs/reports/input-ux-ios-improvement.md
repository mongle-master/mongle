# 기록 입력 UX를 iOS 수준으로 — 개선 보고서

> 대상: `frontend/src/routes/record.tsx`(기록 작성), 부수적으로 `components/person/person-form.tsx`(사람 등록 게이트)
> 기준 문서: [PRD 04-record](../../../docs/prd/04-record.md)
> 작성일: 2026-07-08

---

## 0. 한 줄 요약

몽글은 **"30초 안에, 감정만 골라도 남기는"** 가벼운 기록 앱을 지향하지만(PRD §1),
현재 작성 화면은 **필수 2개(사람·날짜)를 포함한 8개 입력 섹션이 한 번에 평면 나열된 롱 폼**이라
사용자에게 "다 채워야 한다"는 인지 부담을 준다.
핵심 처방은 **① 최소 진입(감정/한 줄) 중심으로 좁히고 나머지는 점진 공개**,
**② 스마트 기본값·자동 제목으로 빈칸을 미리 채우고**,
**③ iOS 네이티브 입력기 최적화(키보드·오토포커스·즉시 저장·초안)** 세 가지다.

## 1. 현재 입력 구조 (실측)

기록 1건을 남길 때 화면에 노출되는 입력 그룹은 **8개**다. 이 중 실제 저장 필수는 **사람·날짜 2개**뿐이고(PRD §4 "최소 저장 조건"), 나머지 6개는 선택인데도 시각적으로 동등하게 나열된다.

| #   | 필드                | UI                         | 필수             | 근거(코드)           |
| --- | ------------------- | -------------------------- | ---------------- | -------------------- |
| 1   | 함께한 사람         | 버튼 → 모달(다중)          | **필수**         | `record.tsx:364-414` |
| 2   | 만남 태그(카테고리) | 토글 칩 단일               | 자동(기본 첫 칩) | `record.tsx:416-436` |
| 3   | 제목                | text input                 | 선택             | `record.tsx:443-449` |
| 4   | 왜                  | textarea                   | 선택             | `record.tsx:460-466` |
| 5   | 무엇을              | textarea                   | 선택             | `record.tsx:473-479` |
| 6   | 언제 — 날짜         | `input[type=date]`         | 필수(기본 오늘)  | `record.tsx:487-492` |
| 7   | 언제 — 시각         | `input[type=time]`         | 선택             | `record.tsx:493-498` |
| 8   | 사진                | `input[type=file]`(최대 5) | 선택             | `record.tsx:502-557` |

**구조적 특징**

- 단일 롱 폼, 스텝/점진 공개 없음 — 8개 섹션이 세로 스크롤로 나열(`record.tsx:341-568`).
- 저장은 명시적 수동 저장(헤더 "저장" 버튼) — **자동저장·초안 없음**(`record.tsx:587-612`, PRD §6에서 초안은 `[post-MVP]`).
- 진입 시 사람 선택 모달을 **닫을 수 없는 상태로 강제 오픈**(`record.tsx:159-174`, `dismissible=false`).
- 폼 라이브러리(react-hook-form/zod)가 설치돼 있으나 미사용 — 필드별 개별 `useState` + 저장 시 일괄 수동 검증(`record.tsx:63-74`, `lib/record-validation.ts`).

## 2. PRD 목표 대비 간극 (진단)

PRD의 목표는 명확하다: **"30초 안에 완료"**, **"감정 하나만 골라도, 아무것도 적지 않아도 남길 수 있을 만큼 부담을 낮춘다"**(PRD §1, §5). 현재 구현은 이 목표에서 다음과 같이 벗어나 있다.

### 2-1. 필드 위계가 없다 — 선택과 필수가 시각적으로 동등

실필수는 사람·날짜뿐인데 8개 섹션이 같은 무게로 나열되어, 사용자는 6개의 선택 필드를 "미완성"으로 인식한다. PRD가 말하는 "세 줄이면 충분해요"라는 카피(`record.tsx:148`)와 화면의 실제 밀도가 어긋난다.

### 2-2. 점진적 공개(progressive disclosure)가 없다

가장 가벼운 기록(감정 하나 + 한 줄)과 가장 무거운 기록(제목·왜·무엇을·날씨·사진·시각)이 **동일한 화면 분량**을 차지한다. iOS의 기록 앱(예: 캘린더 새 이벤트, 미리알림)은 "제목 한 줄 → 필요하면 세부"로 접어둔다.

### 2-3. PRD와의 기능 드리프트 — 오히려 필드가 더 늘어날 예정

- **감정·날씨 칩이 UI에 없다.** PRD §2·§4는 감정(다중)·날씨(단일)를 핵심 입력으로 규정하지만, 코드는 `emotionChipIds: []`, `weatherChipId: null`로 **하드코딩**한다(`record.tsx:239-240`). 즉 "감정만 골라도 저장"이라는 핵심 컨셉이 **아직 구현되지 않았고**, 구현하면 필드가 2개 더 붙는다. → 지금 위계를 잡지 않으면 롱 폼이 10개로 늘어난다.
- **자동 제목이 없다.** PRD §4는 미입력 시 자동 제목 = `{사람} · {카테고리}`를 규정하지만, 코드는 `title.trim() || null`만 보낸다(`record.tsx:233`). 사용자가 제목을 "비워도 되는 칸"으로 인지하지 못한다.

### 2-4. 진입 마찰 — 닫을 수 없는 강제 모달

작성 화면에 들어오자마자 사람 선택 모달이 `dismissible=false`로 뜬다(`record.tsx:166`). iOS 감각에서 "화면을 열자마자 막힌 모달"은 강한 마찰이다. PRD §4는 "사람 프로필·타임라인에서 들어오면 화면을 열자마자 저장 가능"을 지향하는데, ＋진입 시 경험이 이와 반대다.

### 2-5. 안전망 부재 — 자동저장·이탈 확인 없음

작성 중 이탈하면 내용이 사라진다. PRD §6은 "작성 중인 내용이 있어요. 저장하지 않고 나갈까요?" 확인을 요구하지만 미구현. 초안 임시저장은 `[post-MVP]`로 미뤄져 있다.

### 2-6. 네이티브 입력기 최적화 부재

`Input`/`Textarea` 공통 프리미티브(`components/ui/input.tsx`)는 스타일 래퍼일 뿐,
`inputMode`·`enterKeyHint`·`autoCapitalize`·`autoFocus` 등 **iOS 키보드 경험을 좌우하는 속성이 전혀 지정돼 있지 않다.** 모바일에서 이 속성 유무가 "네이티브다움"의 절반을 결정한다.

### 2-7. 사람 등록이라는 무거운 선행 게이트

기록하려면 사람이 1명 이상 등록돼 있어야 하는데, 사람 등록 폼 자체가 이름·생일(연·월·일)·처음 만난 날·관계 태그·좋아하는 것·조심할 것·사진 등 **10개 이상 필드**다(`person-form.tsx:53-68`). "빠르게 한 줄 남기려다 무거운 인물 등록에 막히는" 구조.

## 3. iOS 수준의 원칙 (설계 기준)

Apple HIG의 입력 설계를 이 앱에 대응시키면:

1. **기본값으로 시작한다.** 빈 폼이 아니라 "이미 그럴듯하게 채워진 상태"에서 출발한다(오늘 날짜·기본 카테고리·자동 제목).
2. **점진적으로 드러낸다.** 처음엔 감정 + 한 줄만. 세부는 "더 적기"로 접는다.
3. **직접 조작 · 즉시 반영.** 탭 한 번으로 감정이 선택되고, 저장은 손가락이 닿기 쉬운 곳에.
4. **되돌릴 수 있게, 잃지 않게.** 자동 초안 저장 + 이탈 확인 + 저장 후 실행 취소(undo) 토스트.
5. **키보드가 곧 UI다.** 필드마다 알맞은 키보드/리턴키/자동대문자, 진입 시 첫 필드 오토포커스.
6. **엄지 하나로.** 주요 액션은 하단, 스크롤 최소화, 한 손 도달 범위 안에.

## 4. 개선 방안 (우선순위별)

### P0 — 최소 진입 + 점진 공개로 구조 재편 (가장 큰 효과)

**"감정 + 한 줄"을 1차 화면으로, 나머지는 접는다.**

```
┌─────────────────────────────┐
│  ‹        새 기록        저장 │   ← 저장은 사람 1명만 있으면 항상 활성
│                             │
│  [아바타] 오늘 서연이랑      │   ← 자동 제목이 이 인사를 겸함
│          어땠어요?          │
│                             │
│  😊 🥰 😐 😔 😢  + 감정      │   ← PRD 핵심: 감정만 골라도 저장
│                             │
│  ┌───────────────────────┐  │
│  │ 오늘을 한 줄로 남겨보세요 │  │   ← 큰 단일 입력(왜/무엇을 통합 진입)
│  └───────────────────────┘  │
│                             │
│  ⌄ 더 적기 (사진·날짜·태그)  │   ← 나머지 5개 섹션은 여기 접힘
└─────────────────────────────┘
```

- 위 접힘 영역(disclosure)에 제목·왜/무엇을 분리·날씨·시각·사진·날짜 변경을 넣는다.
- "감정 하나만 골라도 저장"이 화면 상단에서 즉시 가능해진다 → PRD §1 목표 직결.
- 구현: PRD가 요구하는 **감정·날씨 칩을 먼저 구현**(현재 하드코딩 `record.tsx:239-240` 해제)하되, 감정은 1차 화면, 날씨는 접힘 영역에 배치.

### P0 — 스마트 기본값과 자동 제목

- **자동 제목 구현**(PRD §4): 제목 미입력 시 `{대표 이름}{ 외 N명} · {카테고리}`를 클라이언트에서 생성해 placeholder가 아니라 실제 인사/카드 제목으로 노출. 사용자는 제목 칸을 "비워도 되는 것"으로 자연히 인지.
- 카테고리 기본 `만남` 고정(PRD §4), 날짜 기본 오늘 유지 → **빈칸 0개 상태로 진입**.

### P0 — 강제 모달 제거, 인라인 사람 선택

- 진입 시 `dismissible=false` 모달 자동 오픈(`record.tsx:159-174`)을 제거.
- 대신 상단에 **최근/즐겨찾기 사람 아바타 줄**을 인라인으로 깔고 탭으로 선택. "더 찾기"만 모달로.
- ＋진입이어도 화면이 막히지 않고, 프로필/타임라인 진입(프리셋)과 경험이 일관됨.

### P1 — iOS 네이티브 입력기 최적화

공통 `Input`/`Textarea`에 목적별 속성을 부여한다.

| 필드                 | 권장 속성                                                                             |
| -------------------- | ------------------------------------------------------------------------------------- |
| 제목·한 줄·왜·무엇을 | `enterKeyHint="done"`, `autoCapitalize="sentences"`, 진입 시 첫 필드 `autoFocus`      |
| 텍스트 영역          | 입력 증가에 따라 높이 자동 확장(auto-grow), `maxLength` 도달 시 카운터 노출           |
| 날짜/시각            | iOS 기본 휠 피커 활용 + **"오늘/어제" 퀵칩**을 위에 둬 대부분의 경우 피커를 열지 않게 |
| 저장 버튼            | 키보드 위 액세서리 바 또는 하단 고정으로 엄지 도달                                    |

> 현재 `input.tsx:5-17`에는 이런 속성이 없다. 여기서부터 손대면 전 필드에 일괄 적용된다.

### P1 — 사진 입력을 "한 번에 여러 장 + 붙여넣기"

- 현재 `input[type=file]`은 단일 파일만 받아 5장을 5번 나눠 올려야 한다(`record.tsx:506-515`, `handlePhotoPick`이 `files?.[0]`만 사용).
- `multiple` 허용 + 순차 업로드, 클립보드 붙여넣기 지원 → 한 번의 탭으로 여러 장.

### P1 — 안전망: 자동 초안 + 이탈 확인 + 저장 후 Undo

- 입력값을 디바운스로 localStorage에 자동 저장(초안). 다시 들어오면 복원.
- 입력이 있는 상태로 뒤로가기 시 PRD §6 문구로 확인.
- 저장 직후 "기록했어요 · 실행 취소" 토스트 → 잘못 저장해도 안심.

### P2 — 사람 등록 게이트 슬림화

- **이름만으로 즉시 등록** 경로 제공(빠른 추가). 생일·관계·좋아하는 것 등은 나중에 프로필에서 보강.
- 기록 흐름 안에서 "새 사람" 입력 시 이름만 받고 계속 진행 → 무거운 `person-form`(10+필드)에 튕기지 않음.

### P2 — 폼 인프라 정비 (위 개선을 지속 가능하게)

- 이미 설치된 **react-hook-form + zod**로 전환(현재 미사용). 조건부 노출·자동저장·실시간 검증을 필드별 `useState`(`record.tsx:63-74`)로 관리하는 지금 구조는 위 개선을 넣을수록 손이 많이 간다.
- 라벨+에러+카운터를 묶은 **공통 `Field` 컴포넌트**를 만들어 record/person 폼이 공유(현재 shadcn 프리미티브만 있고 Field 추상화 없음).

## 4.5. 2차 피드백 — 복잡성을 만드는 것 / 개선 필요 (재설계)

P0 구현(감정 우선·점진 공개)을 실제로 만들어 보니, "접기/펼치기 롱 폼"은 여전히 폼처럼 보이고 복잡했다. 일기에 가까운 앱은 **폼이 아니라 몰입형 흐름**이어야 한다. 다음을 반영해 재설계한다.

| #   | 복잡성을 만드는 것                                                             | 개선 방향                                                                                                                                          |
| --- | ------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | "더 적기 (제목·왜·태그·날씨·날짜·사진)" — 괄호 안 항목 나열이 과함             | 괄호 제거. 흐름상 다음 단계로 자연히 넘어가므로 나열이 불필요                                                                                      |
| 2   | 기록 진입 시 앱 헤더(홈·제목·저장)+하단 탭바가 그대로 있음 — 모바일스럽지 않음 | **기록 화면은 몰입형 전체화면.** 화면이 스스로 목적을 외치면 헤더·탭바가 필요 없다. `AppShell` 네비 제거                                           |
| 3   | "더 적기"를 숨김 토글로 처리 — 진행감이 없음                                   | **use-funnel로 단계화.** '완료/다음'을 모든 단계에 상시 제공                                                                                       |
| 4   | '오늘의 감정' 같은 필드 라벨                                                   | 한 화면이 한 목적만 담으면 라벨이 불필요 — 화면 자체가 라벨. 제거/최소화                                                                           |
| 5   | 함께한 사람을 선택 카드+변경+chevron으로 다룸 (보통 1명)                       | 선택 기능 제거. **아이폰 전화번호부 상세처럼 큰 아바타 + 아래 이름**만 표시. 프리셋 진입 전제, ＋무프리셋만 첫 화면에서 전화번호부식 목록으로 지정 |
| 6   | 칩이 작음                                                                      | **칩을 크게** — 손가락 타깃·감성 모두 개선                                                                                                         |
| 7   | '무엇을'이 밋밋한 textarea                                                     | **편지지 같은 배경**의 감성적 작성 영역                                                                                                            |

### 재설계 구조 (확정 · 3차 피드백 반영)

- **퍼널 단계(한 화면 한 목적):** `감정` → `무엇을`(편지지) → `세부`(종류·왜·언제).
  - **감정:** 제목 `{이름}님에 대해 기록할게요.` + `오늘은 ___다` 문장. 칩은 과거 서술형(기뻤·감사했·편안했)에 파스텔 컬러. 활용은 규칙화가 어려워 매핑으로 처리(es-hangul은 조사만 지원).
  - **무엇을:** 흑백 편지지(삐뚤빼뚤·옅은 괘선), placeholder `오늘 있었던 일`, 우하단 `n/120` 카운터, 아래 큰 사진 추가 영역. **제목 입력은 없애고 본문 첫 줄을 잘라 제목으로**(엔터=첫 줄까지).
  - **세부:** 질문 헤딩·아바타 없음. `종류`(구 만남 태그) · `언제`(날짜 스트립 + 시간 휠). 날씨 제거. (`왜`는 main의 memo 통합으로 대응 필드가 없어 제거)
- **버튼:** 하단 여백 없이 풀블리드로 상/하 배치. primary(위) `더 자세히 쓰기`(다음 단계), ghost(아래) `다음에 쓰기`(지금 저장). 마지막 단계는 `완료`만.
- **사람:** 감정 단계에 프리셋 사람을 큰 아바타로만 표시(선택 UI 없음). ＋무프리셋 진입만 첫 화면을 전화번호부식 목록으로.
- **레이아웃:** 앱 헤더/하단 탭바 제거(몰입형). 상단 크롬은 `뒤로 · {이름}님 · 완료`(진행 점 제거, `완료`=저장 상시). 하단은 `이어서`(다음) 풀블리드.
- **설명 문구 제거:** "여러 개 골라도…", "여기는 다 건너뛰어도…" 등 보조 설명은 모두 뺀다.
- **서체:** 손글씨(온글잎 은별, `--font-hand`)는 **유저 입력·감정 텍스트에만**(편지지·감정 문장/단어). UI(헤더·라벨·버튼·휠)는 `font-sans`.
- **톤:** 순검정은 눈부심이 있어 `foreground/85` 소프트 잉크로(버튼·선택칩·완료). 배경은 흰색 유지.
- **감정:** 칩 테두리 없이 색상 글자만. **하나만 선택**하고, `오늘은 ___다`의 밑줄 빈칸은 항상 보이며 고른 감정이 그 위에 얹힌다. 큰 아바타(≈62vw)는 감정 단계 상단 — 아바타가 커지면 모노그램 글자도 함께 확대.
- **왜:** main이 왜/무엇을 → `memo` 단일 필드로 통합해, 무엇을 편지지 = memo로 매핑하고 별도 왜 칩은 제거.
- **언제:** 날짜는 **가로 슬라이드 스트립**(과거 30일, 요일+일, 좌로 갈수록 과거), 시간은 **24시간 시/분 휠**(오전·오후 열 없음, 한 번에 5칸, 별도 값 표시 없음, `react-mobile-picker`).
- **저장:** 상단 우측 체크(✓) 아이콘 버튼.
- **사진:** 무엇을 단계에서 편지지 **위**에 큰 추가 영역.
