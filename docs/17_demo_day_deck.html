Problemmongle · 관계도감

관계의 기록은 어디에나 있고,
어디에도 없다

“사람과의 관계 정보가 고정 정보, 추억, 취향, 만남 기록으로 흩어져 있어 한눈에 관계를 이해하기 어렵다.”

  • 감으로 짜맞춘다 — 마지막 만남을 확인하려면 대화방과 사진첩을 일일이 거슬러 올라간다.
  • 알아차릴 계기가 없다 — 소원해진 사람은 그 사실조차 모른 채 그대로 잊힌다.
  • 단서가 사라진다 — ‘그때 친했지’라는 감각만 남고, 왜 가까웠는지는 흐려진다.

다섯 개의 탭, 하나의 도감

홈타임라인＋사람설정
홈나를 중심에 둔 관계 지도. 사람이 노드로, 관계가 선으로 — 흩어져 있던 관계가 한 화면에 모인다.
타임라인나의 통합 연대기. 모든 사람과의 기록이 시간축 하나로 흐른다.
＋ 기록30초 기록. 감정·날씨·카테고리는 칩으로 톡톡. 필수는 ‘함께한 사람’ 하나뿐.
사람인물 도감. 좋아하는 것과 ‘조심할 것’, 생일 D-day, 처음과 마지막으로 만난 날.
설정나만의 어휘. 감정·날씨·카테고리·관계태그 칩을 직접 만들고 고친다.

Productmongle · 관계도감

디테일이 곧 철학

  • 점수를 매기지 않는다 — 관계를 점수화·서열화하는 표현은 전 화면에서 금지. 활동 차트도 횟수 없이 ‘있었다’만 점으로.
  • 알림 대신, 빈 자리 — “남들은 ‘연락하세요’ 알림, 우리는 빈 자리 자체를 보여준다.” 멀어짐은 그 사람의 평소 만남 주기를 기준으로 조용히 흐려질 뿐.
  • 1년 전 오늘 — 같은 날의 기록이 있을 때만 카드가 뜬다. 다시 연락할 핑계를 건네는 방식.
  • 싫어하는 것이 아니라 ‘조심할 것’ — 말투까지 관계를 아끼는 쪽으로.
  • 유채색 0 — 블랙&화이트 에디토리얼. 색으로 사람을 구분하지 않는다. 오늘 이 발표가 흑백인 이유이기도 하다.

How We Used AI — 1mongle · 관계도감

역할 경계를 없앤 릴레이 개발

목표: 프론트가 백엔드 코드를, 백엔드가 프론트 코드를 만진다. 그래서 모두가 하나의 요구사항(PRD)을 보고, 항상 PRD를 먼저 고친 뒤 코드를 고쳤다.

1st

PRD

한 사람이 요구사항을 쓴다. ‘무엇을’만, 구현은 없이.

2nd

1차 코드

FE·BE 각자. 임무는 기능이 아니라 암묵지를 전부 꺼내놓는 것 — 배포 방식, 컨벤션, 기존 코드.

3rd

이어받기

다른 사람이 PRD를 먼저 수정하고, 디자인·기능을 바꾸고 리팩토링한다.

4th

QA

또 다른 사람이 검증한다. 만든 사람은 자기 코드를 못 믿으니까.

실측 — BE 리드가 FE 커밋을, FE 셋이 각자 BE 커밋을 남겼다. “배포 이어하기 핸드오프 프롬프트”가 커밋으로 남아 있다.

  • 아키텍처 규칙을 파일로 성문화
  • 엔티티는 id로만 참조 — JPA 연관관계 금지
  • Swagger 설명 없는 API는 리뷰 반려

commit-msg 훅

  • Conventional Commits 강제
  • 영어 type(scope) + 한국어 제목
  • 형식이 틀리면 커밋 자체가 거부된다

CI

  • lint → format → typecheck → build
  • 사람이 안 봐도 무너지지 않는 바닥

문서를 수명별로 4층으로 쪼갠다

문서와 코드가 어긋나는(drift) 이유는 문서가 코드의 ‘what’을 복사하기 때문. 그래서 층마다 수명을 다르게 가져갔다.

docs/prd‘무엇을’만. 기획자의 고도에서 쓴 단일 진실(SSOT). 코드 수정 전에 항상 먼저 본다.영속
mustpass반드시 지킬 불변식 표. 테스트 코드 대신 “믿고 가는 문서” — 엣지를 AI가 임의 판단하지 못하게 못박는다.영속
docs/temp이번 세션의 구현 계획. 일부러 gitignore — what이 문서에 남는 순간 drift가 시작된다.세션 후 폐기
주석코드로 설명 못 하는 암묵지만. PRD 복붙 금지. 같은 주석이 반복되면 공통화 신호.코드 옆

Issue 하나 크기로 던져라

  • “이 Issue 범위만 구현해” — 스코프 폭주를 막는 한 문장. 이슈 발급도 AI가 일괄로.

포크는 대화로, 디테일은 추천대로

  • 제품 방향이 갈리면 사람과 대화, 사소한 구현은 AI 추천대로. 의사결정에도 계층이 있다.

데모엔 테스트 대신 mustpass

  • 유지비가 속도를 죽이는 구간에선 불변식을 문서로 못박고 믿고 간다. 프로덕션 전환 시 재검토.

세션 계획은 git 밖에

  • 구현 계획이 리포지토리에 남는 순간 문서는 거짓말을 시작한다.

문서는 코드와 가까울수록 좋다

  • 암묵지는 주석으로. 배포는 runbook으로. 규칙은 .claude/rules로 — 전부 코드 옆에.

규칙은 사람이 아니라 훅이 지킨다

  • 커밋의 49%를 AI와 함께 썼지만 히스토리가 균일한 이유.

Nextmongle · 관계도감

다음 층은 이미 설계에 있다

전부 PRD와 백로그에 근거가 남아 있는 확장 경로다.

  • 가져오기(import) — 카톡 대화 .txt, LinkedIn CSV, 캘린더. “직접 내보낸 파일 업로드”가 약관·법을 지키는 유일한 경로라는 분석까지 PRD에 있다.
  • AI 관계 요약·성향 분석 — 기록이 쌓이면 “당신은 오래된 관계를 오래 지키는 사람” 같은 한 문장을 돌려준다.
  • 관계 온도와 재연락 넛지 — ‘멀어짐’ 판정 로직은 이미 구현돼 있다. 온도 곡선은 자연 확장.
  • 공동 기억 — 같은 추억을 서로의 시점에서. 기록이 관계를 다시 잇는 매개가 된다.
  • 나만의 책 — 쌓인 연대기를 물성 있는 아카이브로 묶는다.
