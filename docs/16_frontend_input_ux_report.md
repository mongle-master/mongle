# 프론트엔드 입력 부담 개선 보고서

작성일: 2026-07-08  
대상: `frontend` 기록 작성/사람 등록 입력 UX

## 1. 결론

현재 프론트엔드는 기록앱이라기보다 **관계 CRM에 가까운 폼 구조**로 체감될 가능성이 높다. PRD는 “30초 안에 완료”, “감정만 골라도 저장”을 목표로 잡았지만, 실제 구현은 기록 화면에서 텍스트 입력과 날짜 입력이 중심이고, 사람 등록 화면은 관계 정보까지 한 번에 많이 받는다.

개선 방향은 **정말 필요한 입력만 남기고, 한 번에 하나씩 묻는 퍼널형 흐름**으로 바꾸는 것이다. 현재처럼 여러 입력칸을 한 화면에 펼치면 사용자는 “기록을 남기는 일”보다 “폼을 완성하는 일”로 받아들인다.

추천 원칙은 다음 3가지다.

- **필수 입력 최소화:** 저장을 막는 필수값은 `함께한 사람`만 둔다. 날짜는 오늘, 카테고리는 만남으로 자동 선택한다.
- **퍼널 입력:** 토스처럼 한 화면에서 한 질문만 묻고, 선택하면 다음 단계로 자연스럽게 이동한다.
- **토스식 UX Writing:** 짧고 구체적인 질문, 사용자의 말에 가까운 선택지, 부담을 낮추는 스킵 문구를 쓴다.

## 3. 문제 정의

### 3.1 기록 화면이 “저장”보다 “작성”을 먼저 요구한다

제목, 왜, 무엇을이 모두 선택 입력이어도 화면에서 차지하는 비중이 커서 사용자는 적어야 한다고 느낀다. 일기형 기록앱에서는 빈 입력칸이 많을수록 “나중에 해야지”로 밀릴 가능성이 높다.

### 3.2 가장 쉬운 입력인 감정 칩이 없다

PRD와 우선순위 문서 모두 감정 칩을 핵심으로 둔다. 하지만 현재 구현에서는 감정 값이 빈 배열로 저장된다. 사용자가 타이핑 없이 남길 수 있는 가장 쉬운 기록 단위가 빠져 있다.

### 3.3 새 사람 등록이 기록 흐름을 끊는다

하단 `+`로 기록을 시작했는데 등록된 사람이 없거나 새 사람을 추가해야 하면 전체 사람 추가 화면으로 이동한다. 이 화면은 이름 외 정보까지 많이 보여주므로, “지금 기록”의 맥락이 끊긴다.

### 3.4 선택 입력의 기본값이 UI에 충분히 드러나지 않는다

카테고리는 payload에서 첫 번째 칩을 기본값처럼 쓰지만, UI 상태는 `categoryChipId`가 `null`일 수 있다. 사용자는 기본 선택이 되어 있는지 알기 어렵다. iOS급 편의성은 사용자가 판단하지 않아도 되는 기본 상태를 명확히 보여줘야 한다.

### 3.5 질문이 아니라 항목명으로 쓰여 있다

현재 화면은 `제목`, `왜`, `무엇을`, `언제`, `사진`처럼 폼 라벨 중심이다. 토스식 퍼널 UX에서는 사용자가 답해야 할 질문을 먼저 보여줘야 한다. 예를 들어 `왜`보다 `왜 기억하고 싶어요?`, `무엇을`보다 `무슨 일이 있었어요?`가 낫다. 다만 이 질문도 기본 화면에 모두 펼치면 부담이 커지므로, 단계별로 하나씩만 보여줘야 한다.

## 4. 개선 목표

기록 작성의 목표 시간을 아래처럼 나눈다. 핵심은 모든 사용자가 상세 기록까지 가는 것이 아니라, 대부분의 사용자가 **최소 기록에서 저장을 끝낼 수 있어야 한다**는 점이다.

| 단계       | 목표 시간 | 사용자가 하는 일              | 저장 가능 여부 |
| ---------- | --------: | ----------------------------- | -------------- |
| 최소 기록  |     5-8초 | 사람 확인 또는 선택           | 가능           |
| 감정 기록  |    8-15초 | 감정 칩 1-2개 선택            | 가능           |
| 한 줄 기록 |   15-25초 | 한 줄 메모 입력 또는 건너뛰기 | 가능           |
| 상세 보강  | 30초 이후 | 사진, 날짜, 시간, 상세 문답   | 선택           |

필수 입력은 아래처럼 정리한다.

| 입력                | 저장 전 필수 | 기본값/처리           | 노출 방식                  |
| ------------------- | ------------ | --------------------- | -------------------------- |
| 함께한 사람         | 예           | 진입 경로 사람 프리셋 | 1단계 질문                 |
| 날짜                | 아니오       | 오늘                  | 상세에서만 변경            |
| 카테고리            | 아니오       | 만남                  | 기본값 표시, 상세에서 변경 |
| 감정                | 아니오       | 없음                  | 2단계 칩                   |
| 한 줄 메모          | 아니오       | 없음                  | 3단계, 건너뛰기 가능       |
| 제목/왜/무엇을 분리 | 아니오       | 입력하지 않음         | 상세에서만 표시            |
| 사진                | 아니오       | 없음                  | 저장 후 보강 또는 상세     |
| 시간                | 아니오       | 없음                  | 상세에서만 추가            |

핵심은 **첫 단계에서 이미 저장 가능해야 한다**는 점이다.

## 5. 제안 UX

### 5.1 한 화면 폼을 퍼널 플로우로 바꾼다

기록 작성은 한 화면에 모든 필드를 펼치는 방식이 아니라, 아래 순서의 퍼널로 구성한다.

1. 누구와 함께했는지
2. 어떤 기분이었는지
3. 한 줄로 남길지
4. 저장 완료
5. 더 남길지

각 단계는 뒤로 가기, 건너뛰기, 바로 저장을 제공한다. 사용자가 감정이나 메모를 입력하지 않아도 저장은 막지 않는다.

### 5.2 정말 필요한 입력만 남긴다

MVP 기준 필수 입력은 `함께한 사람` 하나다. 기록앱에서 날짜, 카테고리, 제목, 본문을 모두 필수처럼 보이게 하면 기록 빈도가 떨어진다.

권장 입력 정책:

- `함께한 사람`: 필수. 단, 사람 프로필/타임라인에서 진입하면 자동 선택한다.
- `날짜`: 오늘로 자동 저장한다. 사용자가 바꾸고 싶을 때만 노출한다.
- `카테고리`: `만남`으로 자동 저장한다. 화면에는 작은 보조 텍스트로만 보여준다.
- `감정`: 추천 입력이지만 필수는 아니다.
- `한 줄 메모`: 선택 입력이다. `건너뛰기`가 항상 보여야 한다.
- `사진`: 저장 후 추가할 수 있게 한다.
- `왜/무엇을`: 기본 플로우에서는 제거하고, 상세 기록에서만 쓴다.

이 방식이면 사용자는 “모든 칸을 채워야 한다”는 압박 없이 기록을 시작할 수 있다.

### 5.3 퍼널 단계별 권장 UX Writing

토스식 UX Writing은 설명을 길게 하지 않고, 사용자가 지금 해야 할 행동을 짧게 말한다. 라벨보다 질문, 기능명보다 사용자의 목적을 먼저 쓴다.

| 단계 | 화면 질문             | 선택지/입력                  | 보조 문구                                 | CTA         |
| ---- | --------------------- | ---------------------------- | ----------------------------------------- | ----------- |
| 1    | `누구와 함께했나요?`  | 최근 사람, 즐겨찾기, 새 사람 | `한 명만 골라도 돼요`                     | `다음`      |
| 2    | `오늘 어땠나요?`      | 감정 칩                      | `건너뛰어도 저장돼요`                     | `다음`      |
| 3    | `한 줄로 남겨볼까요?` | 한 줄 입력                   | `나중에 더 써도 돼요`                     | `저장`      |
| 완료 | `기록했어요`          | 요약 카드                    | `사진이나 자세한 이야기를 더할 수 있어요` | `더 남기기` |
| 상세 | `조금 더 남겨볼까요?` | 사진, 날짜, 시간, 왜/무엇을  | `필요한 것만 추가해요`                    | `완료`      |

권장 문구 원칙:

- `제목` 대신 `한 줄로 남겨볼까요?`
- `왜` 대신 `왜 기억하고 싶어요?`
- `무엇을` 대신 `무슨 일이 있었어요?`
- `사진` 대신 `사진도 남길까요?`
- `만남 태그` 대신 `어떤 기록인가요?`
- 에러 문구는 탓하지 않고 해결 행동을 말한다. 예: `함께한 사람을 골라주세요`

피해야 할 문구:

- `입력해주세요`
- `필수 항목입니다`
- `등록하기`
- `상황 기록 작성`
- `만남 태그`

### 5.4 감정 칩을 최우선 구현한다

감정 칩은 지금 빠진 기능 중 가장 먼저 넣어야 한다. 이유는 세 가지다.

- PRD상 “가벼운 기록의 핵심”이다.
- 우선순위 문서에서 기록 작성 1순위다.
- 텍스트 입력 없이도 기록의 의미가 생긴다.

권장 동작:

- 다중 선택
- 최대 5개
- 0개도 저장 가능
- 선택한 감정은 저장 버튼 근처에 요약한다. 예: `민지 · 반가움, 몽글 · 만남`

### 5.5 카테고리는 항상 하나가 선택된 것처럼 보이게 한다

현재 payload는 카테고리 미선택 시 첫 번째 칩을 사용한다. 그렇다면 UI도 첫 번째 칩이 선택된 상태로 시작해야 한다.

권장안:

- `categoryChipId` 초기값을 첫 번째 카테고리 칩으로 잡는다.
- 칩 이름은 기록 화면에서는 `만남 태그`보다 `기록 종류`가 이해하기 쉽다.
- 카테고리는 감정보다 아래에 두거나 제목 입력 오른쪽 드롭다운으로 축소한다.

### 5.6 날짜/시간은 기본값과 빠른 선택으로 처리한다

날짜는 기본 오늘이면 충분하다. 처음 화면에서 `date input`과 `time input`을 나란히 보여주면 사용자가 고쳐야 할 항목으로 인식한다.

권장안:

- 기본 표시는 `오늘` 칩 하나
- 보조 칩: `어제`, `그제`
- 상세에서만 달력 선택
- 시간은 기본 숨김, 필요할 때 `시간 추가`로 연다

### 5.7 사람 선택은 검색 모달보다 최근/즐겨찾기 칩을 먼저 보여준다

현재 사람 선택 모달은 검색과 리스트 중심이다. 등록된 사람이 많아지면 필요하지만, 빠른 기록에서는 최근 사람/즐겨찾기 선택이 더 중요하다.

권장안:

- 기록 화면 상단에 최근 5명/즐겨찾기 사람 칩을 노출
- 현재 선택된 사람은 칩으로 바로 해제/추가
- `변경`을 누르면 기존 모달 진입
- 모달 안에도 검색보다 “최근”, “즐겨찾기” 그룹을 먼저 둔다

### 5.8 새 사람은 “이름만 추가”로 기록 흐름 안에서 만든다

기록 중 새 사람을 추가할 때 전체 프로필 폼으로 이동시키지 않는다.

권장 흐름:

1. 사람 선택 모달에서 `새 사람`
2. 바텀시트 또는 인라인 입력으로 이름만 입력
3. 생성 즉시 선택 상태로 기록 화면 복귀
4. 저장 후 프로필에서 생일, 처음 만난 날, 좋아하는 것 등을 보강

사람 추가 화면 자체도 기본은 이름만 보이고, 나머지는 `프로필 더 채우기`로 접는 편이 낫다.

### 5.9 사람 등록의 필수값을 PRD와 맞춘다

현재 새 사람 등록은 `requireFirstMetYear` 때문에 처음 만난 날의 연도가 필요하다. 기록앱의 빠른 진입을 위해서는 이름만 필수여야 한다.

권장안:

- 새 사람 등록 필수: 이름
- 선택: 사진, 즐겨찾기, 관계 유형
- 접힘: 생일, 처음 만난 날, 관계 태그, 좋아하는 것, 조심할 것

처음 만난 날은 관계 회고에는 유용하지만, 첫 기록을 남기는 순간에는 필수로 두면 안 된다.

## 6. 추천 화면 구조

```text
Step 1
```

## 7. 실행 순서

### P0. 바로 해야 할 일

1. 기록 작성 화면을 퍼널 구조로 바꾼다.
2. 저장 전 필수 입력을 `함께한 사람` 하나로 제한한다.
3. 기록 화면에 감정 칩을 추가하고 `emotionChipIds`에 저장한다.
4. 카테고리와 날짜는 기본값으로 저장하고, 기본 플로우에서는 숨긴다.
5. 제목/왜/무엇을/사진/시간은 `더 남기기`로 보낸다.
6. 새 사람 등록에서 `처음 만난 날` 필수 요구를 제거한다.
7. 사람 선택 모달에 “이름만 새 사람 추가”를 넣는다.
8. 화면 문구를 토스식 질문형 문구로 교체한다.

### P1. 다음 개선

1. 날짜/시간 입력을 `오늘/어제/직접 선택` 패턴으로 바꾼다.
2. 최근 사람/즐겨찾기 사람 칩을 1단계에 둔다.
3. 저장 완료 화면에서 기록 요약을 보여준다.
4. 저장 후 사진/상세 메모를 이어서 추가할 수 있게 한다.
5. 사람 추가 화면도 이름 입력 후 `더 채우기` 퍼널로 바꾼다.

### P2. 나중에 볼 것

1. 임시저장
2. 사진 메타데이터 기반 날짜 추천
3. AI 요약/자동 분류
4. 그림일기 생성

P2는 편의성에는 도움이 되지만, 현재 문제의 핵심인 “입력칸이 많아 보이는 문제”를 먼저 해결하지 못한다. 지금은 P0/P1이 우선이다.

## 8. 성공 기준

개선 후에는 아래 기준으로 판단한다.

- 사람 프로필에서 진입했을 때 저장까지 8초 이내
- 하단 `+`에서 시작했을 때 기존 사람 선택 후 저장까지 15초 이내
- 새 사람과의 기록도 이름만 입력하고 25초 이내 저장 가능
- 한 화면에 텍스트 입력칸이 1개를 넘지 않음
- 각 단계에서 주 질문이 1개만 보임
- `건너뛰기`를 눌러도 저장 가능한 흐름이 유지됨
- 날짜/카테고리를 건드리지 않아도 오늘의 만남 기록으로 저장됨을 사용자가 알 수 있음

## 9. 최종 권고

이번 단계에서는 “iOS급”을 고급 자동화로 해석하지 않는 편이 좋다. iOS급 편의성의 핵심은 사용자가 앱의 데이터 구조를 이해하지 않아도, 지금 남기고 싶은 관계 기록을 한두 번의 선택으로 저장하는 흐름이다.

따라서 구현 우선순위는 다음으로 고정한다.

1. 필수 입력을 `함께한 사람` 하나로 축소
2. 기록 작성 퍼널화
3. 토스식 UX Writing 적용
4. 감정 칩 저장
5. 이름만으로 사람 추가
6. 상세 입력은 저장 후 보강으로 이동
7. 날짜/카테고리 기본값 명확화

이 순서로 처리하면 현재의 “입력 많은 폼” 체감이 “한 번씩 답하면 끝나는 기록”으로 바뀐다.
