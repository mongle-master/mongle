# 02. 인물 — mustpass

> 목적: 인물 도메인(#21~#31) 코드가 반드시 지켜야 할 불변식을 확정.
> 근거: (PRD 02-person §4 필드 · §5 즐겨찾기 · §7 엣지 · README §12.3 글자수 · §12.4 날짜 · §12.5 문구) = SSOT.
> 공통 계약(에러 포맷·소유 컨텍스트·소프트삭제·한도·이미지)은 [00-infra.md](00-infra.md) 를, 관계태그 칩은 [01-chip.md](01-chip.md) 를 그대로 소비한다.
>
> 이 파일은 **코어**(#21 #22 #23 #24 #26 #28 #29 #31)만 다룬다. 상세 조회(#25)·삭제(#27)·파생 스탯(#30)은 event 도메인 확정 후 append 한다.

## 필드 · 소유 (must, #21)

- 인물은 **소유자(ownerId, 데모 사용자)** 에 귀속되고 **소프트삭제**(`deletedAt`)를 상속한다 — 지워도 과거 참조는 남기는 규약([00-infra](00-infra.md)).
- 필드: 이름(필수)·생일(연도 선택)·처음 만난 날·마지막 만난 날·프로필 사진(경로 1장)·관계 유형(짧은 텍스트)·관계 태그(칩 다중)·좋아하는 것·조심할 것·즐겨찾기. 이름 외 전부 선택.
- **즐겨찾기 기본값은 꺼짐(false).**
- **관계 유형은 칩이 아니다** — 한 사람당 하나의 짧은 텍스트(관계 태그와 다른 개념). 최대 20자.

## 생일 연도-선택 (must, #21)

- 생일은 통째로 선택. 넣을 땐 **월·일이 함께** 있어야 하고(둘 중 하나만은 불가), **연도는 생략 가능**(연도 없이 `월 일`만).
- 표현: `birthYear`(nullable) + `birthMonth` + `birthDay`. 월·일 둘 다 null = 생일 없음.

## 글자수 (must, §12.3)

| 대상 | 한도 | 초과 시 |
|---|---|---|
| 이름 | 20자(`NAME_MAX`) | 400 `LENGTH_EXCEEDED` "최대 20자까지 쓸 수 있어요." |
| 관계 유형 | 20자(`RELATION_TYPE_MAX`) | 400 `LENGTH_EXCEEDED` |
| 좋아하는 것 / 조심할 것 (각 항목) | 30자(`PREFERENCE_ITEM_MAX`) | 400 `LENGTH_EXCEEDED` |

- 이름이 비면(공백 포함) 저장하지 않고 400 `REQUIRED_FIELD` "이름을 입력해 주세요.".
- 이름은 유일할 필요 없음(동명이인 허용).

## 날짜 검증 (must, #31 · §12.4)

- **처음 만난 날 ≤ 오늘.** 미래면 400 `FUTURE_DATE`.
- **마지막 만난 날: 처음 만난 날 이후 그리고 오늘 이하.** 처음 만난 날보다 앞서면 400 `DATE_ORDER`, 미래면 400 `FUTURE_DATE`.
  - 처음 만난 날이 없으면 순서 검사는 건너뛴다(마지막 만난 날 미래 검사는 유지).
- **생일:** 연도가 있으면 그 날짜가 미래일 수 없다(400 `FUTURE_DATE`). 연도가 없으면 미래 검사 없음(월·일만으로는 미래 판단 안 함).

## 관계 태그 (must, #22 · §12.2)

- 인물↔관계태그(`RELATION_TAG` 칩) 다대다, **칩 id 참조**(라벨 복사 저장 안 함 — 칩 이름을 바꾸면 저절로 반영).
- 인물당 **최대 10개**(`RELATION_TAG_PER_PERSON_MAX`). 초과면 400 `SELECTION_LIMIT` "관계 태그는 최대 10개까지 담을 수 있어요."(PRD 02 §7).
- 요청에 같은 칩 id 가 중복되면 **첫 등장 기준 1건으로 정규화**해 저장한다(개수 상한 판단도 정규화 이후 개수 기준).
- 존재하지 않거나 **다른 종류**(감정·날씨·카테고리)의 칩 id, 또는 내가 못 보는 칩(타인 개인 칩)은 거부 → 404 `NOT_FOUND`.
- 인물 조회 시 관계 태그를 **포함**한다(홈 그룹화·필터 근거). 라벨은 칩에서 해석 — 소프트삭제된 칩도 라벨은 보인다([00-infra](00-infra.md)).

## 좋아하는 것 · 조심할 것 (must, #23)

- 취향 목록 2종(좋아하는 것 / 조심할 것), 각각 여러 항목.
- 항목당 최대 30자, 각 목록 **최대 20개**(`PREFERENCE_LIST_MAX`). 초과면 400 `SELECTION_LIMIT` "최대 20개까지 담을 수 있어요."(PRD 02 §7).
- **같은 목록 안 중복 금지** → 409 `DUPLICATE`. (좋아하는 것과 조심할 것 사이 중복은 막지 않는다.)
- 기본값은 빈 목록.

## 등록 (must, #24)

- 필수는 **이름뿐**. 나머지(생일·처음/마지막 만난 날·프로필 사진·관계 유형·관계 태그·취향·즐겨찾기)는 모두 선택.
- 날짜(#31)·글자수(§12.3)·관계 태그(#22)·취향(#23) 규칙을 모두 통과해야 저장한다.
- 프로필 사진은 미리 업로드된 경로(url) 1장을 참조한다(업로드는 `POST /api/v1/images`, [00-infra](00-infra.md)).

## 수정 (must, #26)

- 등록과 **같은 입력·같은 검증**을 재사용하는 전체 수정(PUT). 관계 태그·취향 목록은 보낸 값으로 **교체**한다.
- 내 소유가 아니거나 없는 인물은 404 `NOT_FOUND`.

## 즐겨찾기 토글 (must, #28)

- on/off **토글**(별도 확인 없음), 기본 꺼짐. 내 소유·존재하는 인물만, 아니면 404 `NOT_FOUND`.

## 디렉토리 목록 (must, #29)

- 정렬: **이름순(기본)** / **최근 만남순**. 어느 정렬이든 **즐겨찾기는 항상 상단 그룹**.
  - 이름순: 이름 오름차순(대소문자 무시).
  - 최근 만남순: `마지막 만난 날` 최신 먼저, 없는 사람은 뒤로. (이후 event 반영은 파생 단계에서 이 값이 갱신됨.)
- **검색:** 이름·관계 유형 **부분 일치**(대소문자 무시). 비면 전체.
- 내 소유·active(소프트삭제 제외) 인물만 대상.

## 파생 스탯 (must, #30)

- **조회 시 계산**한다(저장 안 함) — 수정·삭제에도 값이 일관되게. 소프트삭제된 기록은 제외. 계산은 `PersonStatsService.statsOf(person)`.
- **만난 횟수** = 카테고리가 `만남`인 그 인물 연결 기록의 **고유 날짜 수**(같은 날 여러 건은 1회).
- **새긴 기록 수** = 그 인물 연결 **전체** 기록 수(카테고리 무관, active 만).
- **마지막 만남** = `max(person.lastMetDate, 만남 카테고리 기록의 최신 날짜)`. 둘 다 없으면 없음(`—`). 수기 입력값과 이벤트를 조회 시 다시 합산하므로, 이벤트가 과거로 수정·삭제돼도 되돌아간다(전진만 하던 `updateLastMetIfNewer` 의 한계를 조회에서 보정).
- **파생 전진(`updateLastMetIfNewer`)은 처음 만난 날보다 앞선 기록 날짜를 무시**한다 — 파생 갱신이 사용자가 확정한 처음 만난 날을 침범하지 않는다(불변식 '마지막 ≥ 처음' 유지).
- **알고 지낸 기간** = 처음 만난 날부터 오늘까지 **가장 큰 단위 1개**(예 `6년`). 처음 만난 날 없으면 없음.
- **경과일(N일째)** = 만난 날을 1일째로 센다(§12.4). **상대시간·경과일·알고 지낸 기간 포맷은 `common/DateDisplay` 로 중앙화**(타임라인 #44 공유).
- **만남 앵커** = 공통 `만남` CATEGORY 칩(`ChipService.meetingCategoryId()`) — rename·삭제 불가라 안정적. 앵커가 없으면(방어) 만남 관련 스탯은 0.
- **재사용:** `statsOf(...).meetingDatesDesc`(만남 고유 날짜, 최신 먼저)가 만난 횟수·마지막 만남·**만남 주기(홈 #41 친밀도)**의 단일 근거다.

## 상세 조회 (must, #25)

- `GET /api/v1/persons/{id}` — 내 소유·active 만, 아니면 404 `NOT_FOUND`.
- 응답 = 기본 정보(이름·생일·처음 만난 날·프로필 사진·관계 유형·즐겨찾기) + 관계 태그(id+라벨) + 취향(좋아하는 것/조심할 것) + **파생 스탯 섹션**(#30).
- 상세의 `lastMetDate` 는 **재계산값**(수기+이벤트 max)을 내려, 표시 truth 를 단일화한다. (디렉토리 목록의 정렬용 `lastMetDate` 는 저장 필드 그대로 — 별개 관심사.)
- 스탯 섹션 필드: 만난 횟수·새긴 기록 수·경과일(N일째)·알고 지낸 기간(표시 문자열)·마지막 만남 상대시간. 값이 없으면 null(프론트가 해당 행을 숨김 — §7).
- **최근 함께한 일(최대 3개)** 은 타임라인(#44) 소관이라 상세 응답에 **포함하지 않는다**(Event→응답 변환을 타임라인 에이전트가 소유 — 결합 회피). 프로필 화면은 타임라인 피드에서 3개를 가져온다.

## 삭제 (must, #27)

- `DELETE /api/v1/persons/{id}` → 204. **소프트삭제**. 내 소유·active 만, 아니면 404 `NOT_FOUND`.
- **연결된 기록 처리:** 그 인물이 연결된 **active** 기록마다 이 인물을 **연결에서 제거**하고, 그 결과 **연결 인물이 0명이 되면**(마지막 연결이었으면) 그 기록도 **소프트삭제**한다.
  - 근거: PRD §7 은 "연결된 기록도 함께 사라진다"만 명시하고 **다인 연결 기록의 처리는 명시하지 않는다.** 데이터 보존 철학([00-infra](00-infra.md): 과거 참조 보존)과 #33 "연결 인물 ≥1" 불변식상, **다른 사람과도 함께한 기록을 그 사람 타임라인에서까지 지우는 통삭제는 부적절**하다 → 연결만 해제, 마지막 연결이면 삭제.
  - 대표(첫 번째) 인물이 제거되면 다음 인물이 대표가 되어 자동 제목(#37)이 조회 시 자연히 갱신된다(id 참조 철학과 일치).
  - 이미 소프트삭제된 기록은 건드리지 않는다(과거 참조 보존).
